38 ### Blink example ###
39 
40 Here is a first example of a FastArduino based program:
41 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
42 #include <fastarduino/gpio.h>
43 #include <fastarduino/time.h>
44 
45 int main()
46 {
47  board::init();
48  sei();
49 
50  gpio::FastPinType<board::DigitalPin::LED>::TYPE led{gpio::PinMode::OUTPUT};
51  while (true)
52  {
53  led.toggle();
54  time::delay_ms(500);
55  }
56  return 0;
57 }
58 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
59 
60 This example can be broken down into several parts:
61 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
62 #include <fastarduino/gpio.h>
63 #include <fastarduino/time.h>
64 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
65 This includes the necessary API from FastArduino; in this example, we just use `gpio.h` (all API for digital input/output) and `time.h` (API for busy loop delays).
66 
67 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
68 int main()
69 {
70  board::init();
71  sei();
72 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
73 The next part defines the standard `main()` function as the entry point of the program; first actions in the `main()` should always be to call `board::init()` (important initialization for some specific boards), then sooner or later call `sei()` to enable interrupts again, as interrupts are disabled when `main()` is initially called.
74 
75 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
76  gpio::FastPinType<board::DigitalPin::LED>::TYPE led{gpio::PinMode::OUTPUT};
77 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
78 This line declares and initializes a digital pin variable named `led` as output for the board's LED (i.e. `D13` on Arduino boards).
79 
80 `board::DigitalPin` is a strong enum class that defines all digital pins **for the current target**, that target must be defined in the compiler command-line.
81 
82 The actual type of `led` is `gpio::FastPin<board::Port:PORT_B, 5>` which means "the pin number 5 within port B"; since this type is not easy to declare when you only know the number of the pin you need, `gpio::FastPinType<board::DigitalPin::LED>::TYPE` is used instead, as it maps directly to the right type.
83 
84 `led` is initialized as an output pin, its initial level is `false` (i.e. GND) by default.
85 
86 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
87  while (true)
88  {
89  led.toggle();
90  time::delay_ms(500);
91  }
92 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
93 Then the program enters an endless loop in which, at every iteration:
94 
95 1. It toggles the level of `led` pin (D13 on Arduino) from GND to Vcc or from Vcc to GND
96 2. It delays execution (i.e. it "waits") for 500 milliseconds
97 
98 This part of the program simply makes your Arduino LED blink at 1Hz frequency!
99 
100 The last part below is never executed (because of the endless loop above) but is necessary to make the compiler happy, as `main()` shall return a value:
101 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
102  return 0;
103 }
104 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
105 
106 Congratulation! We have just studied the typical "blink" program.
107 
108 At this point, it is interesting to compare our first program with the equivalent with standard Arduino API:
109 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
110 void setup()
111 {
112  pinMode(LED_BUILTIN, OUTPUT);
113 }
114 
115 void loop()
116 {
117  digitalWrite(LED_BUILTIN, HIGH);
118  delay(500);
119  digitalWrite(LED_BUILTIN, LOW);
120  delay(500);
121 }
122 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
123 Granted that the latter code seems simpler to write! However, it is also simpler to write it wrong, build and upload it to Arduino and only see it not working:
124 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
125 #define LED 53
126 void setup()
127 {
128  pinMode(LED, OUTPUT);
129 }
130 
131 void loop()
132 {
133  digitalWrite(LED, HIGH);
134  delay(500);
135  digitalWrite(LED, LOW);
136  delay(500);
137 }
138 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
139 The problem here is that Arduino API accept a simple number when they need a pin, hence it is perfectly possible to pass them the number of a pin that does not exist, as in the faulty code above: this code will compile and upload properly to an Arduino UNO, however it will not work, because pin 53 does not exist!
140 
141 This problem cannot occur with FastArduino as the available pins are stored in a strong enum and it becomes impossible to select a pin that does not exist for the board we target!
142 
143 Now, what is really interesting in comparing both working code examples is the size of the built program (measured with UNO as a target, FastArduino project built with AVR Toolchain 3.5.3, Arduino API project built with Arduino IDE 1.8.2):
144 | | Arduino API | FastArduino |
145 |-----------|-------------|-------------|
146 | code size | 928 bytes | 154 bytes |
147 | data size | 9 bytes | 0 byte |
148 
149 As you probably know, Atmel AVR MCU (and Arduino boards that embed them) are much constrained in code and data size, hence we could say that "every byte counts". In the table ablove, one easily sees that Arduino API comes cluttered with lots of code and data, even if you don't need it; on the other hand, FastArduino is highly optimized and will produce code only for what you **do** use.

151 ### LED Chaser example ###
152 
153 Now `gpio.h` has more API than just `gpio::FastPin` and `gpio::FastPinType`; it also includes `gpio::FastPort` and `gpio::FastMaskedPort` that allow to manipulate several pins at a time, as long as these pis belong to the same Port of the MCU. This allows size and speed optimizations when having to deal with a group of related pins, e.g. if you want to implement a LED chaser project.
154 
155 With FastArduino, here is a program showing how you could implement a simple 8 LED chaser on UNO:
156 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
157 #include <fastarduino/gpio.h>
158 #include <fastarduino/time.h>
159 
160 int main()
161 {
162  board::init();
163  sei();
164 
165  gpio::FastPort<board::Port::PORT_D> leds{0xFF, 0x00};
166  uint8_t pattern = 0x01;
167  while (true)
168  {
169  leds.set_PORT(pattern);
170  time::delay_ms(250);
171  pattern <<= 1;
172  if (!pattern) pattern = 0x01;
173  }
174  return 0;
175 }
176 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
177 In this example, we selected all pins of the same port to connect the 8 LEDs of our chaser. Concretely on UNO, this is port D, which pins are D0-D7.
178 
179 We thus declare and initialize `leds` as a `gpio::FastPort<board::Port::PORT_D>` port, with all pins as output (`0xFF`), with initial level to GND (`0x00`, all LEDs off).
180 
181 Then, we will keep track of the current lit LED through `pattern` byte which each bit represents actually one LED; `pattern` is initialized with `0x01` i.e. D0 should be the first LED to be ON.
182 
183 In the endless loop that follows, we perform the following actions:
184 
185 1. Set all pins values at once to the current value of `pattern`
186 2. Delay execution for 250ms
187 3. Shift the only 1 bit of `pattern` left; note that after 8 shifts, `pattern` will become `0`, hence we need to check against this condition to reset `pattern` to its initial state.
188 
189 This should be rather straightforward to understand if you know C or C++.
190 
191 Here is an equivalent example with Arduino API:
192 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
193 const byte LED_PINS[] = {0, 1, 2, 3, 4, 5, 6, 7};
194 const byte NUM_LEDS = sizeof(LED_PINS) / sizeof(LED_PINS[0]);
195 
196 void setup()
197 {
198  for(byte i = 0; i < NUM_LEDS; i++)
199  pinMode(LED_PINS[i], OUTPUT);
200 }
201 
202 void loop()
203 {
204  for(byte i = 0; i < NUM_LEDS; i++)
205  {
206  digitalWrite(LED_PINS[i], HIGH);
207  delay(250);
208  digitalWrite(LED_PINS[i], LOW);
209  }
210 }
211 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
212 We see, with Arduino API, that we have to deal with each pin individually, which makes the program source code longer and not necessarily easier to understand.
213

216 |-----------|-------------|-------------|
217 | code size | 968 bytes | 168 bytes |
218 | data size | 17 bytes | 0 byte |

224 ### Simple Serial Output example ###
225 
226 Although not often necessary in many finished programs, `UART` (for serial communication interface) is often very useful for debugging a program while it is being developed; this is why `UART` is presented now.
227 
228 Here is a first simple program showing how to display, with FastArduino API, a simple string to the serial output (for UNO, this is connected to USB):
229 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
230 #include <fastarduino/uart.h>
231 
232 static constexpr const uint8_t OUTPUT_BUFFER_SIZE = 64;
233 static char output_buffer[OUTPUT_BUFFER_SIZE];
234 
235 REGISTER_UATX_ISR(0)
236 
237 int main() __attribute__((OS_main));
238 int main()
239 {
240  board::init();
241  sei();
242 
243  serial::hard::UATX<board::USART::USART0> uart{output_buffer};
244  uart.register_handler();
245  uart.begin(115200);
246 
247  streams::ostream out = uart.out();
248  out.write("Hello, World!\n");
249  out.flush();
250  return 0;
251 }
252 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
253 As usual, at first we need to include the proper header (`uart.h`) to use its API.
254 
255 Then, we define a buffer that will be used by the `UART` API to transmit characters to your PC through USB. You may find it cumbersome to do it yourself but it brings a huge advantage: you are the one to decide of the buffer size, whereas in Arduino API, you have no choice. Here, we consider 64 bits to be big enough to store characters that will be transmitted to the PC. How `UART` is using this buffer is not important to you though.
256 
257 Then we *register an ISR* necessary for transmissions to take place; this is done by the `REGISTER_UATX_ISR(0)` macro. Explicit ISR registration is one important design choice of FastArduino: **you** decide which ISR should be registered to do what. This may again seem cumbersome but once again this gives you the benefit to decie what you need, hence build your application the way you want it.
258 
259 The code that follows instantiates a `uart::hard::UATX` object that is using `board::USART::USART0` (the only one available on UNO) and based on the previously created buffer. Note that `UATX` class is in charge of **only** transmitting characters, not receiving. Other classes exist for only receiving (`UARX`), or for doing both (`UART`).
260 
261 Once created, `uart` needs to be *linked* to the ISR previously registered, this is done through `uart.register_handler()`. Then we can set `uart` ready for transmission, at serial speed of 115200 bps.
262 
263 Next step consists in extracting, from `uart`, a `streams::ostream` that will allow us to send characters or strings to USB:
264 
265  out.write("Hello, World!\n");
266 
267 The last important instruction, `out.flush()`, waits for all characters to be transmitted before leaving the program.
268 
269 Do note the specific `main` declaration line before its definition: `int main() __attribute__((OS_main));`. This helps the compiler perform some optimization on this function, and may avoid generating several dozens code instructions in some circumstances. In some situations though, this may increase code size by a few bytes; for your own programs, you would have to compile with and without this line if you want to find what is the best for you.
270 
271 Here is the equivalent code with Arduino API:
272 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
273 void setup()
274 {
275  Serial.begin(115200);
276  Serial.println("Hello, World!");
277 }
278 
279 void loop()
280 {
281 }
282 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
283 Of course, we can see here that the code looks simpler, although one may wonder why we need to define a `loop()` function that does nothing.
284 
285 Now let's compare the size of both:
286 | | Arduino API | FastArduino |
287 |-----------|-------------|-------------|
288 | code size | 1440 bytes | 662 bytes |
289 | data size | 200 bytes | 91 bytes |
290 The data size is big because the buffer used by `Serial` has a hard-coded size (you cannot change it without modifying and recompiling Arduino API). Moreover, when using `Serial`, 2 buffers are created, one for input and one for output, even though you may only need the latter one!
291 
292 Now let's take a look at the 91 bytes of data used in the FastArduino version of this example, how are they broken down?
293 | Source | data size |
294 |-------------------|-------------|
295 | `output_buffer` | 64 bytes |
296 | power API | 1 byte |
297 | `UATX` ISR | 2 bytes |
298 | `UATX` vtable | 8 bytes |
299 | "Hello, World!\n" | 16 bytes |
300 | **TOTAL** | 91 bytes |
301 
302 *vtable* is specific data created by C++ compiler for classes with `virtual` methods: every time you use virtual methods in classes, this will add more data size, this is why FastArduino avoids `virtual` as much as possible.
303 
304 As you can see in the table above, the constant string `"Hello, World!\n"` occupies 16 bytes of data (i.e. AVR SRAM) in addition to 16 bytes of Flash (as it is part of the program and must eb stored permanently). If your program deals with a lot of constant strings like this, you may quickly meet a memory problem with SRAM usage. This is why it is more effective to keep these strings exclusively in Flash (you have no choice) but load them t SRAM only when they are needed, i.e. when they get printed to `UATX` as in the sample code.
305 
306 How do we change our program so that this string is only stored in Flash? We can use FastArduino `flash` API for that, by changing only one line of code:
307 
308  out.write(F("Hello, World!\n"));
309 
310 Note the use of `F()` macro here: this makes the string reside in Flash only, and then it is being read from Flash "on the fly" by `out.write()` method; the latter method is overloaded for usual C-strings (initial example) and for C-strings stored in Flash only.
311 
312 We can compare the impact on sizes:
313 | | without %F() | with %F() |
314 |-----------|--------------|-------------|
315 | code size | 662 bytes | 668 bytes |
316 | data size | 91 bytes | 75 bytes |
317 Although a bit more code has been added (the code to read the string from Flash into SRAM on the fly), we see 16 bytes have been removed from data, this is the size of the string constant.
318 
319 You may wonder why `"Hello, World!\n"` occupies 16 bytes, although it should use only 15 bytes (if we account for the terminating `'\0'` character); this is because the string is stored in Flash and Flash is word-addressable, not byte-addressable on AVR.
320 
321 Note that Flash can also be used to store other read-only data that you may want to access at runtime at specific times, i.e. data you do not want to be stored permanently on SRAM during all execution of your program.
322 
323 The following example shows how to:
324 - define, in your source code, read-only data that shall be stored in Flash memory
325 - read that data when you need it
326 
327 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
328 #include <fastarduino/flash.h>
329 
330 // This is the type of data we want to store in flash
331 struct Dummy
332 {
333  uint16_t a;
334  uint8_t b;
335  bool c;
336  int16_t d;
337  char e;
338 };
339 
340 // Define 2 variables of that type, which will be stored in flash
341 // Note the PROGMEM keyword that says the compiler and linker to put this data to flash
342 const Dummy sample1 PROGMEM = {54321, 123, true, -22222, 'z'};
343 const Dummy sample2 PROGMEM = {12345, 231, false, -11111, 'A'};
344 
345 // The following function needs value of sample1 to be read from flash
346 void read_and_use_sample1()
347 {
348  // value will get copied with sample1 read-only content
349  Dummy value;
350  // request reading sample1 from flash into local variable value
351  flash::read_flash(&sample1, value);
352  // Here we can use value which is {54321, 123, true, -22222, 'z'}
353 
354 }
355 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

357 ### Formatted Output example ###
358 
359 Compared to Arduino API, FastArduino brings formatted streams as can be [found in standard C++](https://en.wikipedia.org/wiki/Input/output_(C%2B%2B)); although more verbose than usual C `printf()` function, formatted streams allow compile-time safety.
360 
361 Here is an example that prints formatted data to USB:
362 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
363 #include <fastarduino/uart.h>
364 
365 static constexpr const uint8_t OUTPUT_BUFFER_SIZE = 64;
366 static char output_buffer[OUTPUT_BUFFER_SIZE];
367 
368 REGISTER_UATX_ISR(0)
369 
370 using namespace streams;
371 
372 int main()
373 {
374  board::init();
375  sei();
376 
377  serial::hard::UATX<board::USART::USART0> uart{output_buffer};
378  uart.register_handler();
379  uart.begin(115200);
380 
381  ostream out = uart.out();
382  uint16_t value = 0x8000;
383  out << F("value = 0x") << hex << value
384  << F(", ") << dec << value
385  << F(", 0") << oct << value
386  << F(", B") << bin << value << endl;
387  return 0;
388 }
389 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
390 Here, we still use `uart.out()`, but here we use its "insertion operator" `<<`.
391 
392 If you are used to programming with C++ for more usual systems (e.g. Linux), then you will immediately recognize [`std::ostream` API](http://www.cplusplus.com/reference/ostream/ostream/operator%3C%3C/) which FastArduino library tries to implement with some level of fidelity.
393 
394 You can also find more details in `streams` namespace documentation.
395 
396 Here is the equivalent code with Arduino API:
397 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
398 void setup()
399 {
400  Serial.begin(115200);
401  byte value = 0x8000;
402  Serial.print(F("value = 0x"));
403  Serial.print(value, 16);
404  Serial.print(F(", "));
405  Serial.print(value);
406  Serial.print(F(", 0"));
407  Serial.print(value, 8);
408  Serial.print(F(", B"));
409  Serial.println(value, 2);
410 }
411 
412 void loop()
413 {
414 }
415 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

420 | code size | 1808 bytes | 1804 bytes |

423 ### Serial Input example ###
424 
425 FastArduino also implements input streams connected to serial output; here is a simple example:
426 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
427 #include <fastarduino/uart.h>
428 
429 static constexpr const uint8_t INPUT_BUFFER_SIZE = 64;
430 static char input_buffer[INPUT_BUFFER_SIZE];
431 
432 REGISTER_UARX_ISR(0)
433 
434 int main()
435 {
436  board::init();
437  sei();
438 
439  serial::hard::UARX<board::USART::USART0> uart{input_buffer};
440  uart.register_handler();
441  uart.begin(115200);
442 
443  streams::istream in = uart.in();
444 
445  // Wait until a character is ready and get it
446  char value;
447  in.get(value);
448 
449  // Wait until a complete string is ready and get it
450  char str[64+1];
451  in.get(str, 64+1);
452 
453  return 0;
454 }
455 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
456 Note the similarities between this example and UATX example above for all the setup parts.
457 The main differences are:
458 - use `UARX` type instead of `UATX`
459 - `REGISTER_UARX_ISR()` instead of `REGISTER_UATX_ISR()` macro for ISR registration
460 - use `istream` instead of `ostream` and `uart.in()` instead of `uart.out()`
461 
462 Then `UARX` mainly offers one method, `get()`, which returns the next character serially received and buffered; if the input buffer is currently empty, then `get()` will block.
463 
464 The example uses 2 flavors of this `istream` method:
465 - `get(char&)`: this method **blocks** until one character is available on serial input.
466 - `get(char*, size_t, char)`: this blocks until a complete string (terminated by a specified delimiter character, `'\n'` by default) gets read on serial input and fills the given buffer parameter with that string content.
467 
468 Note that these 2 methods use `time::yield()` while waiting; this may be linked to `power` management. Please take a look at the documentation for this API for further details.

470 ### Formatted Input example ###
471 
472 Similar to output, input streams supports formatted input, as can be found in standard C++; once again, formatted input streams allow compile-time safety.
473 
474 The following example uses formatted input to read values from USB:
475 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
476 #include <fastarduino/uart.h>
477 
478 // Define vectors we need in the example
479 REGISTER_UARX_ISR(0)
480 
481 // Buffers for UARX
482 static const uint8_t INPUT_BUFFER_SIZE = 64;
483 static char input_buffer[INPUT_BUFFER_SIZE];
484 
485 using INPUT = streams::istream;
486 
487 int main()
488 {
489  board::init();
490  sei();
491 
492  // Start UART
493  serial::hard::UARX<board::USART::USART0> uarx{input_buffer};
494  uarx.register_handler();
495  uarx.begin(115200);
496  INPUT in = uarx.in();
497 
498  // Wait for a char
499  char value1;
500  in >> streams::skipws >> value1;
501 
502  // Wait for an uint16_t
503  uint16_t value2;
504  in >> streams::skipws >> value2;
505 
506  return 0;
507 }
508 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
509 Here, we still use `uart.in()` but we use its "extraction operator" `>>`. All extractions are blocking and will not return until the required type can be read from the buffer.
510 
511 If you are used to programming with C++ for more usual systems (e.g. Linux), then you will immediately recognize [`std::istream` API](http://www.cplusplus.com/reference/istream/istream/operator%3E%3E/) which FastArduino library tries to implement with some level of fidelity.
512 
513 You can also find more details in `streams` namespace documentation.
514 
515 We have already seen `UATX` and `UARX` as classes for sending, resp. receiving, data through serial. There is also `UARX` which combines both.
516 
517 As you know, the number of physical (hardware) UART available on an MCU target is limited, some targets (ATtiny) don't even have any hardware UART at all. For this reason, if you need extra UART featurs to connect to some devices, you can use software UART API, documented in [namespace `serial::soft`](TODO). As this is more complicated to use, it is not part of this basic tutorial, but will be addressed later on.

520 @anchor analoginput Basics: analog input
521 ----------------------------------------
522 
523 Here is a simple example using analog input API to read a value from some sensor (thermistor, potentiometer, whatever you like) and lights a LED if the read value is above some threshold:
524 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
525 #include <fastarduino/analog_input.h>
526 #include <fastarduino/gpio.h>
527 #include <fastarduino/time.h>
528 
529 const uint16_t THRESHOLD = 500;
530 
531 int main()
532 {
533  board::init();
534  sei();
535 
536  gpio::FastPinType<board::DigitalPin::LED>::TYPE led{gpio::PinMode::OUTPUT};
537  analog::AnalogInput<board::AnalogPin::A0> sensor;
538  while (true)
539  {
540  if (sensor.sample() > THRESHOLD)
541  led.set();
542  else
543  led.clear();
544  time::delay_ms(100);
545  }
546  return 0;
547 }
548 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
549 This example is an adaptation of the [first GPIO example](@ref gpio) of this tutorial.
550 
551 The first change consists in including the necessary header:
552 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
553 #include <fastarduino/analog_input.h>
554 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
555 
556 Then we have the definition of the `sensor` variable:
557 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
558  analog::AnalogInput<board::AnalogPin::A0> sensor;
559 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
560 Here we instantiate `AnalogInput` for analog pin `A0` (on Arduino UNO).
561 
562 In the infinite loop, we then get the current analog value of `sensor` and compare it to `THRESHOLD` constant:
563 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
564  if (sensor.sample() > THRESHOLD)
565 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
566 
567 By default, sample values are on 10 bits (0..1023) represented as `uint16_t`.
568 
569 If you don't need such precision, you can define `sensor` differently:
570 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
571  analog::AnalogInput<board::AnalogPin::A0, board::AnalogReference::AVCC, uint8_t> sensor;
572 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
573 Note the two additional template arguments provided to `AnalogInput<...>`:
574 - the first added argument `board::AnalogReference::AVCC`, although seldom changed, may become important when you create your own boards from MCU chips; you can further read API documentation if you need more information about it
575 - the second added argument is the type of returned samples, either `uint16_t` (default value) or `uint8_t`. The type determines the samples precision:
576  - `uint8_t`: 8 bits (samples between 0 and 255)
577  - `uint16_t`: 10 bits (samples between 0 and 1023)

607 @anchor timer Basics: timer
608 ---------------------------
609 A timer (it should actually be named "timer/counter") is a logic chip or part of an MCU that just "counts" pulses of a clock at a given frequency. It can have several modes. It is used in many occasions such as:
610 - real time counting
611 - asynchronous tasks (one-shot or periodic) scheduling
612 - PWM signal generation (see [PWM](@ref pwm) for further details)
613 
614 A timer generally counts up, but it may also, on some occasions, count down; it may trigger interrupts on several events (i.e. when counter reaches some specific limits), it may drive some specific digital output pins, and sometimes it may also be driven by digital input pins (to capture counter value).
615 
616 There are typically several independent timers on an MCU, but they are not all the same. Timers may differ in:
617 - counter size (8 or 16 bits for AVR timers)
618 - list of settable frequencies (timer frequencies are derived from the MCU clock by prescaler devices)
619 - the timer modes supported
620 - the pins they are connected to
621 - specific capabilities they may have
622 
623 Rather than explaining the theory further, we will start studying a simple example that uses a timer for blinking a LED, a little bit like the first example in this tutorial, but totally driven asynchronously:
624 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
625 #include <fastarduino/gpio.h>
626 #include <fastarduino/timer.h>
627 
628 constexpr const board::Timer TIMER = board::Timer::TIMER1;
629 using CALCULATOR = timer::Calculator<TIMER>;
630 using TIMER_TYPE = timer::Timer<TIMER>;
631 constexpr const uint32_t PERIOD_US = 1000000;
632 
633 constexpr const TIMER_TYPE::TIMER_PRESCALER PRESCALER = CALCULATOR::CTC_prescaler(PERIOD_US);
634 constexpr const TIMER_TYPE::TIMER_TYPE COUNTER = CALCULATOR::CTC_counter(PRESCALER, PERIOD_US);
635 
636 class Handler
637 {
638 public:
639  Handler(): _led{gpio::PinMode::OUTPUT, false} {}
640 
641  void on_timer()
642  {
643  _led.toggle();
644  }
645 
646 private:
647  gpio::FastPinType<board::DigitalPin::LED>::TYPE _led;
648 };
649 
650 // Define vectors we need in the example
651 REGISTER_TIMER_COMPARE_ISR_METHOD(1, Handler, &Handler::on_timer)
652 
653 int main() __attribute__((OS_main));
654 int main()
655 {
656  board::init();
657  sei();
658  Handler handler;
659  interrupt::register_handler(handler);
660  TIMER_TYPE timer{timer::TimerMode::CTC, PRESCALER, timer::TimerInterrupt::OUTPUT_COMPARE_A};
661  timer.begin(COUNTER);
662 
663  while (true) ;
664 }
665 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
666 This example looks much more complex than all previous examples but it is straightforward to understand once explained part after part.
667 
668 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
669 #include <fastarduino/gpio.h>
670 #include <fastarduino/timer.h>
671 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
672 In addition to GPIO, we include the header containing all Timer API.
673 
674 For this example, we use Arduino UNO, which MCU (ATmega328P) includes 3 timers (named respectively `Timer0`, `Timer1`, `Timer2` in its datasheet), we use Timer1 which is 16-bits in size:
675 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
676 constexpr const board::Timer TIMER = board::Timer::TIMER1;
677 using CALCULATOR = timer::Calculator<TIMER>;
678 using TIMER_TYPE = timer::Timer<TIMER>;
679 constexpr const uint32_t PERIOD_US = 1000000;
680 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
681 Although not needed, it is a good practice to define a `const`, named `TIMER` in this snippet, valued with the real timer we intend to use.
682 Then we define 2 new type aliases, `CALCULATOR` and `TIMER` that will help us type less code (this is common recommended practice when using C++ templates heavily in programs):
683 - `CALCULATOR` is the type of a class which provides `static` utility methods that will help us configure the timer we have selected; do note that, since all timers are different, `CALCULATOR` is specific to one timer only; hence if our program was using 2 distinct timers, we would have to define two distinct calculator type aliases, one for each timer.
684 - `TIMER_TYPE` is the type of the class that embed all timer API for the specific timer we have selected.
685 
686 Finally we define `PERIOD_US` the period, in microseconds, at which we want the LED to blink. Please note that this is in fact half the actual period, because this is the time at which we will toggle the LED light.
687 
688 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
689 constexpr const TIMER_TYPE::TIMER_PRESCALER PRESCALER = CALCULATOR::CTC_prescaler(PERIOD_US);
690 constexpr const TIMER_TYPE::TIMER_TYPE COUNTER = CALCULATOR::CTC_counter(PRESCALER, PERIOD_US);
691 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
692 The above snippet defines constant settings, computed by `CALCULATOR` utility class, that we will later use to initialize our timer:
693 - `PRESCALER` is the **optimum** prescaler value that we can use for our timer in order to be able to count up to the requested period, i.e. 1 second; the type of prescaler is an `enum` that depends on each timer (because the list of available prescaler values differ from one timer to another). The prescaler defines the number by which the MCU clock frequency will be divided to provide the pulses used to increment the timer. We don't need to know this value or fix it ourselves because `CALCULATOR::CTC_prescaler` calculates the best choice for us.
694 - `COUNTER` is the maximum counter value that the timer can reach until 1 second has ellapsed; its type is based on the timer we have selected (i.e. `Timer1` => 16 bits => `uint16_t`), but we don't need to fix this type ourselves because it depends on the timer we have selected.
695 
696 Note that, although we know in advance which timer we use, we always avoid declaring direct types (such as `uint16_t`) in order to facilate a potential change to another timer in the future, without having to change several code locations.
697 
698 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
699 class Handler
700 {
701 public:
702  Handler(): _led{gpio::PinMode::OUTPUT, false} {}
703 
704  void on_timer()
705  {
706  _led.toggle();
707  }
708 
709 private:
710  gpio::FastPinType<board::DigitalPin::LED>::TYPE _led;
711 };
712 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
713 Here we define the class which implements the code in charge of blinking the LED every time the timer has reached its maximum value, i.e. every second.
714 There is nothing special to explain here, except that the method `on_timer()` is a *callback function* which will get called asynchronously (from interrupt handlers) when the timer reaches its max.
715 
716 Since timers generate interruptions, we need to "attach" our handler code above to the suitable interruption, this is done through the following line of code:
717 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
718 REGISTER_TIMER_COMPARE_ISR_METHOD(1, Handler, &Handler::on_timer)
719 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
720 `REGISTER_TIMER_COMPARE_ISR_METHOD` is a macro that will generate extra code (code you do not need, nor want, to see) to declare the Interrupt Service Routine (*ISR*) attached to the proper interruption of our selected timer; it takes 3 arguments:
721 - `1` is the timer number (`0`, `1` or `2` on UNO)
722 - `Handler` is the class that contains the code to be called when the interrupt occurs
723 - `&Handler::on_timer` is the Pointer to Member Function (often abbreviated *PTMF* by usual C++ developers) telling which method from `Handler` shall be called back when the interrupt occurs
724 In FastArduino, interrupt handling follows some patterns that are further described [here](@ref interrupts) and won't be developed in detail now.
725 
726 Now we can finally start writing the code of the `main()` function:
727 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
728 int main() __attribute__((OS_main));
729 int main()
730 {
731  board::init();
732  sei();
733  Handler handler;
734  interrupt::register_handler(handler);
735 
736 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
737 Past the usual initialization stuff, this code performs an important task regarding interrupt handling: it creates `handler`, an instance of the `Handler` class that has been defined before as the class to handle interrupts for the selected timer, and then it **registers** this handler instance with FastArduino. Now we are sure that interrupts for our timer will call `handler.on_timer()`.
738 
739 The last part of the code creates and starts the timer we need in our program:
740 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
741  TIMER_TYPE timer{timer::TimerMode::CTC, PRESCALER, timer::TimerInterrupt::OUTPUT_COMPARE_A};
742  timer.begin(COUNTER);
743 
744  while (true) ;
745 }
746 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
747 `timer` is the instance of `timer::Timer` API for `board::Timer::TIMER1`; upon instantiation, it is passed the timer mode to use, the previously calculated clock prescaler, and the interrupt we want to enable.
748 
749 Here we use *CTC* mode (Clear Timer on Compare); in this mode the counter is incremented until it reaches a maximum value, then it triggers an interrupt and it clears the counter value back to zero and starts counting up again.
750 
751 To ensure that our handler to get called back when the timer reaches 1 second, we set `timer::TimerInterrupt::OUTPUT_COMPARE_A`, which enables the proper interrupt on this timer: when the counter is reached, an interrupt will occur, the properly registered ISR will be called, and in turn it will call our handler.
752 
753 Then `timer.begin()` activates the timer with the maximum counter value, that was calculated initially in the program. This value, along with `PRESCALER`, has been calculated in order for `timer` to generate an interrupt (i.e. call `handler.on_timer()`) every second.
754 
755 Note the infinite loop `while (true);` at the end of `main()`: without it the program would terminate immediately, giving no chance to our timer and handler to operate as expected. What is interesting to see here is that the main code does not do anything besides looping forever: all actual stuff happens asynchronously behind the scenes!
756 
757 I would have liked to perform a size comparison with Arduino API, but unfortunately, the basic Arduino API does not provide an equivalent way to directly access a timer, hence we cannot produce the equivalent code here. Anyway, here is the size for the example above:
758 | | FastArduino |
759 |-----------|-------------|
760 | code size | 248 bytes |
761 | data size | 2 bytes |

764 @anchor rtt Basics: real-time timer
765 -----------------------------------
766 
767 A real-time timer is primarily a device that tracks time in standard measurements (ms, us).
768 
769 It may be used in various situations such as:
770 - delay program execution for some us or ms
771 - capture the duration of some event with good accuracy
772 - implement timeouts in programs waiting for an event to occur
773 - generate periodic events
774 
775 The simple example that follows illustrates the first use case:
776 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
777 #include <fastarduino/gpio.h>
778 #include <fastarduino/realtime_timer.h>
779 
780 REGISTER_RTT_ISR(0)
781 
782 const constexpr uint32_t BLINK_DELAY_MS = 500;
783 
784 int main()
785 {
786  board::init();
787  sei();
788 
789  timer::RTT<board::Timer::TIMER0> rtt;
790  rtt.register_rtt_handler();
791  rtt.begin();
792 
793  gpio::FastPinType<board::DigitalPin::LED>::TYPE led{gpio::PinMode::OUTPUT};
794  while (true)
795  {
796  led.toggle();
797  rtt.delay(BLINK_DELAY_MS);
798  }
799 }
800 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
801 This example looks much like the first blinking example in this tutorial, with a few changes.
802 
803 First off, as usual the neceaary header file is included:
804 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
805 #include <fastarduino/realtime_timer.h>
806 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
807 Then we need to register an ISR for the RTT feature to work properly:
808 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
809 REGISTER_RTT_ISR(0)
810 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
811 Then, in `main()`, after the usual initialization stuff, we create a real-time timer instance, based on AVR UNO Timer0 (8-bits timer), register it with the ISR previously registered with `REGISTER_RTT_ISR(0)` macro, and finally starts it counting time.
812 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
813  timer::RTT<board::Timer::TIMER0> rtt;
814  rtt.register_rtt_handler();
815  rtt.begin();
816 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
817 Finally, we have the usual loop, toogling the LED, and then delay for 10s, using the RTT API:
818 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
819  rtt.delay(BLINK_DELAY_MS);
820 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
821 
822 Let's examine the size of this program and compare it with the first example of this tutorial, which used `time::delay_ms()`:
823 | | delay_ms | RTT::delay |
824 |-----------|------------|-------------|
825 | code size | 154 bytes | 404 bytes |
826 | data size | 0 bytes | 3 bytes |
827 
828 As you can see, code and data size is higher here, so what is the point of using `RTT::delay()` instead of `time::delay_ms()`? The answer is **power consumption**:
829 - `time::delay_ms` is a busy loop which requires the MCU to be running during the whole delay, hence consuming "active supply current" (about 15mA for an ATmega328P at 16MHz)
830 - `RTT::delay()` will set the MCU to pre-defined sleep mode and will still continue to operate well under most available sleep modes (this depends on which timer gets used, refer to [AVR datasheet](TODO) for further details); this will alow reduction of supply current, hence power consumption. Current supply will be reduced more or less dramatically according to the selected sleep mode.
831 
832 Another practical use of RTT is to measure the elapsed time between two events. For instance it can be used with an ultrasonic ranging device to measure the duration of an ultrasound wave to do a roundript from the device to an obstacle, then calculate the actual distance in mm. The following snippet shows how it could look like for an HC-SR04 sensor:
833 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
834 // Declare 2 pins connected to HC-SR04
835 gpio::FastPinType<board::DigitalPin::D0>::TYPE trigger{gpio::PinMode::OUTPUT};
836 gpio::FastPinType<board::DigitalPin::D1>::TYPE echo{gpio::PinMode::INPUT};
837 
838 // Declare RTT (note: don't forget to call REGISTER_RTT_ISR(1) macro in your program)
839 timer::RTT<board::Timer::TIMER1>& rtt;
840 
841 // Send a 10us pulse to the trigger pin
842 trigger.set();
843 time::delay_us(10);
844 trigger.clear();
845 
846 // Wait for echo signal start
847 while (!echo.value()) ;
848 // Reset RTT time
849 rtt.millis(0);
850 // Wait for echo signal end
851 while (echo.value()) ;
852 // Read current time
853 time::RTTTime end = rtt.time();
854 // Calculate the echo duration in microseconds
855 uint16_t echo_us = uint16_t(end.millis * 1000UL + end.micros);
856 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
857 Note that this snippet is just an example and is not usable as is: it does not include a timeout mechanism to avoid waiting the echo signal forever (which can happen if the ultrasonic wave does not encounter an obstacle within its possible range, i.e. 4 meters). Also, this approach could be improved by making it interrupt-driven (i.e. having interrupts generated when the `echo_pin` changes state).
858 
859 Another interesting use of RTT is to perform some periodic actions. FastArduino implements an events handling mechanism that can be connected to an RTT in order to deliver periodic events. This mechanism is [described later](@ref events) in this tutorial.

862 @anchor pwm Basics: PWM
863 -----------------------
864 
865 PWM (*Pulse Width modulation*) is a technique that can be used to simulate generation of an analog voltage level through a purely digital output. This is done by varying the *duty cycle* of a rectangular pulse wave, i.e. the ratio of "on" time over the wave period.
866 
867 PWM is implemented by MCU through timers.
868 
869 FastArduino includes special support for PWM. The following example demonstrates PWM to increase then decrease the light emitted by a LED:
870 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
871 #include <fastarduino/time.h>
872 #include <fastarduino/pwm.h>
873 
874 static constexpr const board::Timer TIMER = board::Timer::TIMER0;
875 using TIMER_TYPE = timer::Timer<TIMER>;
876 using CALC = timer::Calculator<TIMER>;
877 static constexpr const uint16_t PWM_FREQUENCY = 450;
878 static constexpr const TIMER_TYPE::TIMER_PRESCALER PRESCALER = CALC::FastPWM_prescaler(PWM_FREQUENCY);
879 
880 static constexpr const board::DigitalPin LED = board::PWMPin::D6_PD6_OC0A;
881 using LED_PWM = analog::PWMOutput<LED>;
882 
883 int main()
884 {
885  board::init();
886  sei();
887 
888  // Initialize timer
889  TIMER_TYPE timer{timer::TimerMode::FAST_PWM};
890  timer.begin(PRESCALER);
891 
892  LED_PWM led{timer};
893  // Loop of samplings
894  while (true)
895  {
896  for (LED_PWM::TYPE duty = 0; duty < LED_PWM::MAX; ++duty)
897  {
898  led.set_duty(duty);
899  time::delay_ms(50);
900  }
901  for (LED_PWM::TYPE duty = LED_PWM::MAX; duty > 0; --duty)
902  {
903  led.set_duty(duty);
904  time::delay_ms(50);
905  }
906  }
907  return 0;
908 }
909 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
910 The program starts by including the header for PWM API; this will automatically include the timer API header too.
911 
912 Then a timer is selected for PWM (note that the choice of a timer imposes the choice of possible pins) and a prescaler value computed for it, based on the PWM frequency we want to use, 450Hz, which is generally good enough for most use cases (dimming a LED, rotating a DC motor...):
913 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
914 static constexpr const board::Timer TIMER = board::Timer::TIMER0;
915 using TIMER_TYPE = timer::Timer<TIMER>;
916 using CALC = timer::Calculator<TIMER>;
917 static constexpr const uint16_t PWM_FREQUENCY = 450;
918 static constexpr const TIMER_TYPE::TIMER_PRESCALER PRESCALER = CALC::FastPWM_prescaler(PWM_FREQUENCY);
919 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
920 
921 Then we define the pin that will be connected to the LED and the PWMOutput type for this pin:
922 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
923 static constexpr const board::DigitalPin LED = board::PWMPin::D6_PD6_OC0A;
924 using LED_PWM = analog::PWMOutput<LED>;
925 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
926 Note that `board::PWMPin` namespace limits the pins to PWM-enabled pins; also note the pin name `D6_PD6_OC0A` includes useful information:
927 - this is pin `D6` on Arduino UNO
928 - this pin is on `PD6` i.e. Port D bit #6
929 - this pin is connectable to `OC0A` i.e. Timer 0 COM A
930 
931 Then, in `main()`, after the usual initialization code, we initialize and start the timer:
932 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
933  // Initialize timer
934  TIMER_TYPE timer{timer::TimerMode::FAST_PWM};
935  timer.begin(PRESCALER);
936 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
937 Notice we use the *Fast PWM* mode here, but we might as well use *Phase Correct PWM* mode.
938 
939 Next we connect the LED pin to the timer:
940 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
941  LED_PWM led{timer};
942 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
943 
944 In the main loop, we have 2 consecutive loops, the first increases the light, the second decreases it. Both loops vary the duty cycle between its limits:
945 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
946  for (LED_PWM::TYPE duty = 0; duty < LED_PWM::MAX; ++duty)
947  {
948  led.set_duty(duty);
949  time::delay_ms(50);
950  }
951 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
952 Note the use of `LED_PWM::TYPE`, which depends on the timer selected (8 or 16 bits), and `LED_PWM::MAX` which provides the maximum value usable for the duty cycle, i.e. the value mapping to 100% duty cycle. Pay attention to the fact that `LED_PWM::TYPE` is unsigned, this explains the 2nd loop:
953 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
954  for (LED_PWM::TYPE duty = LED_PWM::MAX; duty > 0; --duty)
955  {
956  led.set_duty(duty);
957  time::delay_ms(50);
958  }
959 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
960 Here, we shall not use `duty >= 0` as the `for` condition, because that condition would be always `true`, hence the loop would be infinite.
961 
962 Now let's compare this example with the Arduino API equivalent:
963 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
964 #define LED 6
965

967 }
968

985 - you cannot choose the PWM frequency, this is imposed to you by Arduino API and varies depending on which timer is used

996 -------------------------------
997 
998 FastArduino provides several general utilities, gathered inside one namespace `utils`.
999 
1000 We will not demonstrate each of these utilities here but just show a few of them in action. In your own programs, if you find yourself in need of some helper stuff that you think deserves to be provided as a general utility, then first take a look at FastArduino [utilities API documentation](namespaceutils.html) and check if you don't find it there, or something similar that you could use in your situation.

1004 - value conversion utilities: functions to help convert a value from one referential to another, very useful when dealing with sensors of all sorts
1005 
1006 ### Low-level utilities examples
1007 
1008 The few examples in this section will introduce you to a few functions that may prove useful if you need to handle devices that are not natively supported by FastArduino.
1009 
1010 1. `utils::swap_bytes`: this function is useful whenever you use a sensor device that provides you with integer values, coded on 2 bytes, with high byte first and low byte second; since AVR MCU are "little-endian" processors, they expect words in the opposite order: low byte first, high byte second, hence in order to interpret values provided by that device, you need to first swap their bytes. Bytes swap is performed "in-place", i.e. the original value is replace with the converted value. The following example is an excerpt of `hmc5883l.h` provided by FastArduino, where magnetic fields in 3 axes have to be converted from big endian (as provided by the HMC5883L) to little endian (as expected by the AVR MCU):
1011 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1012  bool magnetic_fields(MagneticFields& fields)
1013  {
1014  if ( this->write(DEVICE_ADDRESS, OUTPUT_REG_1, i2c::BusConditions::START_NO_STOP) == i2c::Status::OK
1015  && this->read(DEVICE_ADDRESS, fields, i2c::BusConditions::REPEAT_START_STOP) == i2c::Status::OK)
1016  {
1017  utils::swap_bytes(fields.x);
1018  utils::swap_bytes(fields.y);
1019  utils::swap_bytes(fields.z);
1020  return true;
1021  }
1022  else
1023  return false;
1024  }
1025 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1026

1047 ### Conversion utilities examples
1048 
1049 Device sensors measure some physical quantity and generally provide you with some integer value that somehow maps to the physical value. hence to make use of the raw value provided by a sensor, you need to convert it to some more meaningful value that you can understand and operate upon.
1050 
1051 Or conversely, you may just need to compare the physical value agains some thresholds (e.g. check the gyroscopic speed according to some axis is less than 10°/s), and perform some action when this is not true. In this situation, you don't really need to convert the raw sensor value into a physical quantity to compare to the physical threshold, but rather convert (once only) the physical threshold into the corresponding raw value (a constant in your program) and then only compare raw values, which is:
1052 - more performant (no conversion needed before comparison)
1053 - more size efficient (conversion of threshold can be done at compile time, hence no code is generated for it)
1054 
1055 FastArduino utilities provide several conversion methods between raw and physical quantities, according to raw and physical ranges (known for each sensor), and unit prefix (e.g. kilo, mega, giga, centi, milli...). These methods are `constexpr`, which means that, when provided with constant arguments, they will be evaluated at compile-time and return a value that is itself stored as a constant.
1056 
1057 1. `utils::map_physical_to_raw`: although it may seem complicated by its list of arguments, this function is actually pretty simple, as demonstrated in the snippet hereafter:
1058 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1059  static constexpr const int16_t ACCEL_1 = map_physical_to_raw(500, UnitPrefix::MILLI, 2, 15);
1060 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1061 In this example, we convert the acceleration value 500mg (g is *9.81 m/s/s*) to the equivalent raw value as produced by an MPU-6050 accelerometer, using *+/-2g* range (`2` is the max physical value we can get with this device using this range) where this raw value is stored on `15 bits` (+1 bit for the sign), i.e. `32767` is the raw value returned by the device when the measured acceleration is `+2g`.
1062 
1063 2. `utils::map_raw_to_physical`: this method does the exact opposite of `utils::map_physical_to_raw` with the same parameters, reversed:
1064 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1065  int16_t rotation = map_raw_to_physical(raw, UnitPrefix::CENTI, 250, 15);
1066 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1067 In this example, we convert `raw` which is returned by the MPU-6050 gyroscope, using range *+/-250°/s* with `15 bits` precision (+1 bit for the sign), i.e. `32767` is the raw value returned by the device when the measured rotation speed is `+250°/s`. The calculated value is returned in *c°/s* (centi-degrees per second).
1068 
1069 In addition to these functions, FastArduino utilities also include the more common `utils::map` and `utils::constrain` which work like their Arduino API equivalent `map()` and `constrain()`.

1141 ### Introduction to AVR interrupts
1142 
1143 Many AVR MCU features are based upon interrupts, so that the MCU can efficiently react to events such as timer overflow, pin level change, without having to poll for these conditions. Using AVR interrupts makes your program responsive to events relevant to you. This alos allows taking advantage of power sleep modes thus reducing energy consumption while the MCU has nothing to do other than wait for events to occur.
1144 
1145 Each AVR MCU has a predefined list of interrupts sources, each individually activable, and each with an associated *vector* which is the address of an Interrupt Service Routine (**ISR**), executed whenver the matching interrupt occurs. Each ISR is imposed a specific name.
1146 
1147 Among common interrupt vectors, you will find for example:
1148 - `TIMERn_OVF_vect`: triggered when Timer *n* overflows
1149 - `INTn_vect`: triggered when input pin *INTn* changes level
1150 - `UARTn_RX_vect`: triggered when a new character has been received on serial receiver *UARTn`
1151 - ...
1152 
1153 ### AVR interrupts handling in FastArduino
1154 
1155 In FastArduino, ISR are created upon your explicit request, FastArduino will never add an ISR without your consent. FastArduino calls this ISR creation a **registration**.
1156 
1157 ISR registration is performed through macros provided by FastArduino. There are essentially 4 flavours of registration macros:
1158 1. API-specific registration: in this flavour, a FastArduino feature requires to directly be linked to an ISR, through a dedicated macro and a specific registration method (either implicitly called by constructor or explicitly through a specific method). In the previous examples of this tutorial, you have already encountered `REGISTER_UATX_ISR()`, `REGISTER_UARX_ISR()`, `REGISTER_UART_ISR()` and `REGISTER_RTT_ISR()`.
1159 All macros in this flavour follow the same naming scheme: `REGISTER_XXX_ISR`, where *XXX* is the feature handled.
1160 2. Empty ISR registration: in this flavour, you activate an interrupt but do not want any callback for it, but then you have to define an empty ISR for it; this empty ISR will not increase code size of your program. You may wonder why you would want to enable an interrupt but do nothing when it occurs, in fact this is often used to make the MCU sleep (low power consumption) and wake it once an interrupt occurs. You have already seen such usage in a previous example, where `REGISTER_WATCHDOG_ISR_EMPTY()` was used.
1161 3. Method callback registration: with this flavour, you activate an interrupt and want a sepcific method of a given class to be called back when the interrupt occurs; in this flavour, a second step is required inside your code: you need to register an instance of the class that was registered. In this tutorial, previous examples used this approach with `REGISTER_TIMER_COMPARE_ISR_METHOD()` macro and `interrupt::register_handler(handler);` instance registration in `main()`. This is probably the most useful approach as it allows to pass an implicit context (`this` class instance) to the callback.
1162 4. Function callback registration: with this flavour, you can register one of your functions (global or static) as a callback of an ISR. This approach does not require an extra registration step. This is not used as often as the Method callback registration flavour above.
1163 
1164 Whenever method callback is needed (typically for flavours 1 and 3 above), then a second registration step is needed.
1165 
1166 All FastArduino API respects some guidelines for naming ISR registration macros. All macros are in one of the following formats:
1167 - `REGISTER_XXX_ISR()` for API-specific registration
1168 - `REGISTER_XXX_ISR_EMPTY()` for empty ISR
1169 - `REGISTER_XXX_ISR_CALLBACK()` for method callback
1170 - `REGISTER_XXX_ISR_FUNCTION()` for function callback

1195 ### Pin Interrupts
1196 
1197 One very common usage of AVR interrupts is to handle level changes of specific digital input pins. AVR MCU have two kinds of pin interrupts:
1198 - External Interrupts: an interrupt can be triggered for a specific pin when its level changes or is equal to some value (high or low); the number of such pins is quite limited (e.g. only 2 on Arduino UNO)
1199 - Pin Change Interrupts: an interrupt is triggered when one in a set of pins has a changing level; the same interrupt (hence the same ISR) is used for all pins (typically 8, but not necessarily), thus the ISR must determine, by its own means, which pin has triggered the interrupt; although more pins support this interrupt, it is less convenient to use than external interrupts.

1464 REGISTER_PCI_ISR_METHOD(PCI_NUM, EventGenerator, &EventGenerator::on_pin_change, board::DigitalPin::D0_PD0)
1465 
1466 void blink(uint8_t buttons)
1467 {
1468  // If no button is pressed, do nothing
1469  if (!buttons) return;
1470 
1471  gpio::FastPinType<LED>::TYPE led;
1472  // Buttons are plit in 2 groups of four:
1473  // - 1st group sets 5 iterations
1474  // - 2nd group sets 10 iterations
1475  // Note: we multiply by 2 because one blink iteration means toggling the LED twice
1476  uint8_t iterations = (buttons & 0x0F ? 5 : 10) * 2;
1477  // In each group, each buttons define the delay between LED toggles
1478  // - 1st/5th button: 200ms
1479  // - 2nd/6th button: 400ms
1480  // - 3rd/7th button: 800ms
1481  // - 4th/8th button: 1600ms
1482  uint16_t delay = (buttons & 0x11 ? 200 : buttons & 0x22 ? 400 : buttons & 0x44 ? 800 : 1600);
1483  while (iterations--)
1484  {
1485  led.toggle();
1486  time::delay_ms(delay);
1487  }
1488 }
1489 
1490 static const uint8_t EVENT_QUEUE_SIZE = 32;
1491 
1492 int main() __attribute__((OS_main));
1493 int main()
1494 {
1495  board::init();
1496 
1497  // Prepare event queue
1498  EVENT buffer[EVENT_QUEUE_SIZE];
1499  containers::Queue<EVENT> event_queue{buffer};
1500 
1501  // Create and register event generator
1502  EventGenerator generator{event_queue};
1503  interrupt::register_handler(generator);
1504 
1505  // Setup PCI interrupts
1506  interrupt::PCISignal<BUTTONS_PORT> signal;
1507  signal.enable_pins_(0xFF);
1508  signal.enable_();
1509 
1510  // Setup LED pin as output
1511  gpio::FastPinType<LED>::TYPE led{gpio::PinMode::OUTPUT};
1512 
1513  // Enable interrupts at startup time
1514  sei();
1515 
1516  // Event Loop
1517  while (true)
1518  {
1519  EVENT event = containers::pull(event_queue);
1520  if (event.type() == BUTTON_EVENT)
1521  // Invert levels as 0 means button pushed (and we want 1 instead)
1522  blink(event.value() ^ 0xFF);
1523  }
1524  return 0;
1525 }
1526 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1527 In this example, 8 buttons are connected to one port and generate Pin Change Interrupts when pressed. The registered
1528 ISR generates an event for each pin change; the main event loop pulls one event from the events queue and starts a
1529 blinking sequence of the LED connected to D13 (Arduino UNO); the selected sequence (number of blinks and delay between
1530 blinks) depends on the pressed button.
1531 
1532 First off, you need to include the heeader with FastArduino Events API:
1533 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1534 #include <fastarduino/events.h>
1535 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1536 
1537 Then we will be using `namepsace events` in our example:
1538 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1539 using namespace events;
1540 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1541 
1542 Events API defines `template<typename T> class Event<T>` which basically contains 2 properties:
1543 - `type()` which is an `uint8_t` value used to discriminate different event types. FastArduino predefines a few event
1544 types and enables you to define your own types as you need,
1545 - `value()` is a `T` instance which you can use any way you need; each event will transport such a value. If you do not need any additional information on your own events, then you can use `void` for `T`.
1546 
1547 In the example, we use events to transport the state of all 8 buttons (as a `uint8_t`):
1548 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1549 using EVENT = Event<uint8_t>;
1550 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1551 
1552 We also define a specific event type when buttons state changes:
1553 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1554 static constexpr const uint8_t BUTTON_EVENT = Type::USER_EVENT;
1555 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1556 
1557 Note that `Type` is a namespace inside `events` namepsace, which lists FastArduino pre-defined Event types as constants:
1558 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1559  const uint8_t NO_EVENT = 0;
1560  const uint8_t WDT_TIMER = 1;
1561  const uint8_t RTT_TIMER = 2;
1562 
1563  // User-defined events start here (in range [128-255]))
1564  const uint8_t USER_EVENT = 128;
1565 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1566 Numbers from 3 to 127 are reserved for FastArduino future enhancements.
1567 
1568 The we define the class that will handle Pin Change Interrupts and generate events:
1569 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1570 class EventGenerator
1571 {
1572 public:
1573  EventGenerator(containers::Queue<EVENT>& event_queue):event_queue_{event_queue}, buttons_{0x00, 0xFF} {}
1574 
1575  void on_pin_change()
1576  {
1577  event_queue_.push_(EVENT{BUTTON_EVENT, buttons_.get_PIN()});
1578  }
1579 
1580 private:
1581  containers::Queue<EVENT>& event_queue_;
1582  gpio::FastPort<BUTTONS_PORT> buttons_;
1583 };
1584 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1585 Note the use of `containers::Queue<EVENT>` that represents a [ring-buffer](https://en.wikipedia.org/wiki/Circular_buffer)
1586 queue to which events are pushed when the state of buttons are changing, and from which events are pulled from the main
1587 event loop (as shown later).
1588 
1589 The most interesting code is the single line:
1590 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1591  event_queue_.push_(EVENT{BUTTON_EVENT, buttons_.get_PIN()});
1592 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1593 where a new `EVENT` is instantiated, its value is the current level of all 8 buttons pins; then the new event is pushed
1594 to the events queue.
1595 
1596 As [usual](@ref interrupts), we have to register this handler to the right PCI ISR:
1597 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1598 REGISTER_PCI_ISR_METHOD(PCI_NUM, EventGenerator, &EventGenerator::on_pin_change, board::DigitalPin::D0_PD0)
1599 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1600 
1601 Next we define the size of the events queue, which must be a power of 2 (this allows code size and speed optimizations):
1602 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1603 static const uint8_t EVENT_QUEUE_SIZE = 32;
1604 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1605 This must be known at compile-time as we do not want dynamic memory allocation in an embedded program.
1606 
1607 The actual events queue is instantiated inside `main()`:
1608 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1609  EVENT buffer[EVENT_QUEUE_SIZE];
1610  containers::Queue<EVENT> event_queue{buffer};
1611 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1612 Note that this is done in 2 steps:
1613 1. A buffer of the required size is allocated (in the stack for this example); the actual size, in this example, is `64`
1614 bytes, i.e. 32 times 1 byte (event type) + 1 byte (event value).
1615 2. A `Queue` is instantiated for this buffer; after this step, you should never directly access `buffer` from your code.
1616 
1617 Finally, the `main()` function has the infinite event loop:
1618 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1619  while (true)
1620  {
1621  EVENT event = containers::pull(event_queue);
1622  if (event.type() == BUTTON_EVENT)
1623  // Invert levels as 0 means button pushed (and we want 1 instead)
1624  blink(event.value() ^ 0xFF);
1625  }
1626 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1627 The call to `containers::pull(event_queue)` is blocking until an event is available in the queue, i.e. has been pushed
1628 by `EventGenerator::on_pin_change()`. While waiting, this call uses `time::yield()` which may put the MCU in sleep mode (see [Power management tutorial later](@ref power)).
1629 
1630 Once an event is pulled from the event queue, we check its type and call `blink()` function with the event value, i.e.
1631 the state of buttons when Pin Change Interrupt has occurred.
1632 
1633 The `blink()` function simply loops for the requested number of iterations and blinking delay, and returns only when the
1634 full blinking sequence is finished. This function is very simple and there is no need to explain it further.
1635 
1636 ### Event Dispatcher
1637 
1638 In the previous example, only one type of event is expected inside the event loop. In more complex applications though,
1639 many more types of events are foreseeable and we may end up with an event loop like:
1640 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1641  while (true)
1642  {
1643  EVENT event = containers::pull(event_queue);
1644  switch (event.type())
1645  {
1646  case EVENT_TYPE_1:
1647  // Do something
1648  break;
1649 
1650  case EVENT_TYPE_2:
1651  // Do something else
1652  break;
1653 
1654  case EVENT_TYPE_3:
1655  // Do yet something else
1656  break;
1657 
1658  ...
1659  }
1660  }
1661 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1662 This generally makes code less readable and more difficult to maintain.
1663 
1664 FastArduino API supports an *event dispatching* approach, where you define `EventHandler` classes and register
1665 instances for specific event types:
1666 
1667 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1668 class MyHandler: public EventHandler<EVENT>
1669 {
1670 public:
1671  MyHandler() : EventHandler<EVENT>{Type::USER_EVENT} {}
1672  virtual void on_event(const EVENT& event) override
1673  {
1674  // Do something
1675  }
1676 };
1677 
1678 int main()
1679 {
1680  ...
1681  // Prepare Handlers
1682  MyHandler handler;
1683  ...
1684 
1685  // Prepare Dispatcher and register Handlers
1686  Dispatcher<EVENT> dispatcher;
1687  dispatcher.insert(handler);
1688  ...
1689 }
1690 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1691 
1692 With `events::Dispatcher`, you can register one handler for each event type, and then the main event loop
1693 is reduced to the simple code below:
1694 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1695  while (true)
1696  {
1697  EVENT event = pull(event_queue);
1698  dispatcher.dispatch(event);
1699  }
1700 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1701 The right event handler gets automatically called for each pulled event.
1702 
1703 Note that `events::EventHandler` uses a `virtual` method, hence this may have an impact on memory size consumed in
1704 Flash (vtables) and also on method call time.

1706 ### Scheduler & Jobs
1707 
1708 Some types of events are generated by FastArduino features. In particular, watchdog and realtime timer features can be
1709 setup to trigger some time events at given periods. From these events, FastArduino offers a `events::Scheduler` API,
1710 that allows you to define periodic `events::Job`s.
1711 
1712 The next code demonstrates a new way of blinking a LED by using watchdog generated events and associated jobs:
1713 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1714 #include <fastarduino/gpio.h>
1715 #include <fastarduino/events.h>
1716 #include <fastarduino/watchdog.h>
1717 #include <fastarduino/scheduler.h>
1718 
1719 using namespace events;
1720 using EVENT = Event<void>;
1721 
1722 // Define vectors we need in the example
1723 REGISTER_WATCHDOG_CLOCK_ISR(EVENT)
1724 
1725 static const uint32_t PERIOD = 1000;
1726 
1727 class LedBlinkerJob: public Job
1728 {
1729 public:
1730  LedBlinkerJob() : Job{0, PERIOD}, led_{gpio::PinMode::OUTPUT} {}
1731  virtual void on_schedule(UNUSED uint32_t millis) override
1732  {
1733  led_.toggle();
1734  }
1735 
1736 private:
1737  gpio::FastPinType<board::DigitalPin::LED>::TYPE led_;
1738 };
1739 
1740 // Define event queue
1741 static const uint8_t EVENT_QUEUE_SIZE = 32;
1742 static EVENT buffer[EVENT_QUEUE_SIZE];
1743 static containers::Queue<EVENT> event_queue{buffer};
1744 
1745 int main() __attribute__((OS_main));
1746 int main()
1747 {
1748  board::init();
1749  // Enable interrupts at startup time
1750  sei();
1751 
1752  // Prepare Dispatcher and Handlers
1753  Dispatcher<EVENT> dispatcher;
1754  watchdog::Watchdog<EVENT> watchdog{event_queue};
1755  watchdog.register_watchdog_handler();
1756  Scheduler<watchdog::Watchdog<EVENT>, EVENT> scheduler{watchdog, Type::WDT_TIMER};
1757  dispatcher.insert(scheduler);
1758 
1759  LedBlinkerJob job;
1760  scheduler.schedule(job);
1761 
1762  // Start watchdog
1763  watchdog.begin(watchdog::WatchdogSignal::TimeOut::TO_64ms);
1764 
1765  // Event Loop
1766  while (true)
1767  {
1768  EVENT event = pull(event_queue);
1769  dispatcher.dispatch(event);
1770  }
1771 }
1772 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1773 
1774 FastArduino `Scheduler` API is defined in its own header, but under the same `events` namespace:
1775 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1776 #include <fastarduino/scheduler.h>
1777 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1778 
1779 This API is based on FastArduino events management, hence we need to define the `Event` type we need:
1780 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1781 using EVENT = Event<void>;
1782 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1783 FastArduino `Scheduler` does not use `Event`'s`value`, hence we can simply use `void`, so that one `EVENT` will be
1784 exactly one byte in size.
1785 
1786 Next step consists in registering Watchdog clock ISR:
1787 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1788 REGISTER_WATCHDOG_CLOCK_ISR(EVENT)
1789 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1790 
1791 We then define a `Job` subclass that will be called back at the required period:
1792 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1793 class LedBlinkerJob: public Job
1794 {
1795 public:
1796  LedBlinkerJob() : Job{0, PERIOD}, led_{gpio::PinMode::OUTPUT} {}
1797  virtual void on_schedule(UNUSED uint32_t millis) override
1798  {
1799  led_.toggle();
1800  }
1801 
1802 private:
1803  gpio::FastPinType<board::DigitalPin::LED>::TYPE led_;
1804 };
1805 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1806 `Job` constructor takes 2 arguments:
1807 - `next`: the delay in ms after which this job will execute for the first time; `0` means this job should execute
1808 immediately
1809 - `period`: the period in ms at which the job shall be re-executed; `0` means this job will execute only once.
1810 
1811 In this example, we set the job period to 1000ms, i.e. the actual blink period will be 2s.
1812 
1813 The `virtual` method `Job::on_schedule()` will get called when the time to executed has elapsed, not necessarily at the
1814 exact expected time (the exact time depends on several factors exposed hereafter).
1815 
1816 In the example, we then declare the event queue, this time as static memory (not in the stack):
1817 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1818 static const uint8_t EVENT_QUEUE_SIZE = 32;
1819 static EVENT buffer[EVENT_QUEUE_SIZE];
1820 static containers::Queue<EVENT> event_queue{buffer};
1821 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1822 
1823 In `main()`, we create a `Dispatcher` that will be used in the event loop, a `Watchdog` that
1824 will generate and queue `Type::WDT_TIMER` events on each watchdog timeout, and a `Scheduler` that will
1825 get notified of `Type::WDT_TIMER` events and will use the created `Watchdog` as a clock, in order to
1826 determine which `Job`s to call.
1827 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1828  Dispatcher<EVENT> dispatcher;
1829  watchdog::Watchdog<EVENT> watchdog{event_queue};
1830  watchdog.register_watchdog_handler();
1831  Scheduler<watchdog::Watchdog<EVENT>, EVENT> scheduler{watchdog, Type::WDT_TIMER};
1832  dispatcher.insert(scheduler);
1833 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1834 Note the last code line, which links the `Scheduler` to the `Dispatcher` so it can handle proper events.
1835 
1836 Next, we instantiate the job that will make the LED blink, and schedule with `Scheduler`:
1837 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1838  LedBlinkerJob job;
1839  scheduler.schedule(job);
1840 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1841 
1842 We can then start the watchdog with any suitable timeout value (the selected value impacts the accuracy of the
1843 watchdog clock):
1844 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1845  watchdog.begin(watchdog::WatchdogSignal::TimeOut::TO_64ms);
1846 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1847 
1848 Finally, then event loop is quite usual:
1849 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1850  while (true)
1851  {
1852  EVENT event = pull(event_queue);
1853  dispatcher.dispatch(event);
1854  }
1855 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1856 
1857 Jobs scheduling accuracy depends on several factors:
1858 - the accuracy of the `CLOCK` used for `Scheduler`; FastArduino provides two possible clocks (but you may also
1859 provide your own): `watchdog::Watchdog` and `timer::RTT`.
1860 - the CPU occupation of your main event loop, which may trigger a `Job` long after its expected time.
1861 
1862 As mentioned above, `Scheduler` can also work with `timer::RTT`, that would mean only a few changes in the previous
1863 example, as shown in the next snippet, only showing new lines of code:
1864 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1865 #include <fastarduino/realtime_timer.h>
1866 ...
1867 static const uint16_t RTT_EVENT_PERIOD = 1024;
1868 REGISTER_RTT_EVENT_ISR(0, EVENT, RTT_EVENT_PERIOD)
1869 ...
1870 void main()
1871 {
1872  ...
1873  timer::RTT<board::Timer::TIMER0> rtt;
1874  rtt.register_rtt_handler();
1875  timer::RTTEventCallback<EVENT, RTT_EVENT_PERIOD> callback{event_queue};
1876  interrupt::register_handler(callback);
1877  ...
1878  Scheduler<timer::RTT<board::Timer::TIMER0>, EVENT> scheduler{rtt, Type::RTT_TIMER};
1879  ...
1880  rtt.begin();
1881  ...
1882 }
1883 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1884 
1885 Since this code uses a realtime timer, two lines of code instantiate, then start one such timer:
1886 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1887  timer::RTT<board::Timer::TIMER0> rtt;
1888  rtt.register_rtt_handler();
1889  ...
1890  rtt.begin();
1891 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1892 
1893 In addition, we use `timer::RTTEventCallback` utility class, which purpose is to generate events from
1894 RTT ticks:
1895 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1896  timer::RTTEventCallback<EVENT, RTT_EVENT_PERIOD> callback{event_queue};
1897  interrupt::register_handler(callback);
1898 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1899 Here we create a callback that will generate an event every `RTT_EVENT_PERIOD` milliseconds.
1900 
1901 We also need to register the right ISR for the RTT and the callback:
1902 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1903 REGISTER_RTT_EVENT_ISR(0, EVENT, RTT_EVENT_PERIOD)
1904 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1905 
1906 Now the scheduler can be instantiated with the RTT instance as its clock and `Type::RTT_TIMER` as
1907 the type of events to listen to:
1908 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1909  Scheduler<timer::RTT<board::Timer::TIMER0>, EVENT> scheduler{rtt, Type::RTT_TIMER};
1910 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

1913 @anchor power Advanced: Power Management
1914 ----------------------------------------
1915 
1916 AVR MCU support various sleep modes that you can use to reduce power consumption of your circuits.
1917 
1918 Each sleep mode has distinct characteristics regarding:
1919 - maximum current consumed
1920 - list of wake-up signals (interrupts)
1921 - time to wake up from sleep
1922 
1923 FastArduino support for AVR power modes is quite simple, gathered in `namespace power` and consists in
1924 a single class `power::Power` with 3 static methods.
1925 
1926 In addition, the available sleep modes for a given MCU are defined in `board::SleepMode` enumerated type.
1927 
1928 Here is a simple example:
1929 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1930 #include <fastarduino/gpio.h>
1931 #include <fastarduino/power.h>
1932 #include <fastarduino/watchdog.h>
1933 
1934 // Define vectors we need in the example
1935 REGISTER_WATCHDOG_ISR_EMPTY()
1936 
1937 int main() __attribute__((OS_main));
1938 int main()
1939 {
1940  board::init();
1941  sei();
1942 
1943  gpio::FastPinType<board::DigitalPin::LED>::TYPE led{gpio::PinMode::OUTPUT};
1944 
1945  watchdog::WatchdogSignal watchdog;
1946  watchdog.begin(watchdog::WatchdogSignal::TimeOut::TO_500ms);
1947 
1948  while (true)
1949  {
1950  led.toggle();
1951  power::Power::sleep(board::SleepMode::POWER_DOWN);
1952  }
1953 }
1954 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
1955 You may have recognized it: this is the example presented in the [Watchdog section](@ref watchdog) of this tutorial.
1956 
1957 In the main loop, we just toggle the LED pin and go to `board::SleepMode::POWER_DOWN` sleep mode (the mode that
1958 consumes the least current of all).
1959 
1960 The `Power::sleep()` method will return only when the MCU is awakened by an external signal, in this example, the
1961 signal used is the watchdog timeout interrupt (every 500ms).
1962 
1963 Note that the `time::yield()` method just calls `Power::sleep()` (with current default sleep mode) and this method
1964 is used by other FastArduino API on several occasions whenever these API need to wait for something:
1965 - `watchdog::Watchdog::delay()` while waiting for time to elapse for the given delay
1966 - `timer::RTT::delay()` while waiting for time to elapse for the given delay
1967 - `streams::ostreambuf::pubsync()` while waiting until all stream content has been read by a consumer (e.g. UART)
1968 - `containers::pull()` and `containers::peek()` while waiting for an item to be pushed to a queue
1969 - `devices::rf::NRF24L01` when sending or receving payload
1970 
1971 Hence when using any of these API, it is important to select the proper sleep mode, based on your power
1972 consumption requirements.

1975 @anchor eeprom Advanced: EEPROM
1976 -------------------------------
1977 
1978 All AVR MCU include an EEPROM (Electrically Erasable Programmable Read-Only Memory) which can be useful for storing
1979 data that:
1980 - is quite stable but may change from time to time
1981 - is fixed but different for every MCU (for a large series of similar circuits)
1982 
1983 Typical examples of use of EEPROM cells include (but are not limited to):
1984 - Circuit unique ID (e.g. used in a network of devices)
1985 - WIFI credentials
1986 - Calibration value (for internal RC clock)
1987 - Sequence of tones frequencies & periods for an alarm melody
1988 
1989 In these examples, writing EEPROM cells would happen only once (or a few times), while reading would occur at startup.
1990 
1991 ### Reading EEPROM content
1992 
1993 Often, writing EEPROM can be done directly through an ISP programmer (UNO USB programming does not support EEPROM writing) and only reading is needed in a program. The next example illustrates this situation.
1994 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
1995 #include <fastarduino/eeprom.h>
1996 #include <fastarduino/tones.h>
1997 #include <fastarduino/time.h>
1998 
1999 // Board-dependent settings
2000 static constexpr const board::Timer NTIMER = board::Timer::TIMER1;
2001 static constexpr const board::DigitalPin OUTPUT = board::PWMPin::D9_PB1_OC1A;
2002 
2003 using devices::audio::Tone;
2004 using namespace eeprom;
2005 using GENERATOR = devices::audio::ToneGenerator<NTIMER, OUTPUT>;
2006 
2007 struct TonePlay
2008 {
2009  Tone tone;
2010  uint16_t ms;
2011 };
2012 
2013 // Melody to be played
2014 TonePlay music[] EEMEM =
2015 {
2016  // Intro
2017  {Tone::A1, 500},
2018  {Tone::A1, 500},
2019  {Tone::A1, 500},
2020  {Tone::F1, 350},
2021  {Tone::C2, 150},
2022  {Tone::A1, 500},
2023  {Tone::F1, 350},
2024  {Tone::C2, 150},
2025  {Tone::A1, 650},
2026 
2027  // Marker for end of melody
2028  {Tone::END, 0}
2029 };
2030 
2031 int main()
2032 {
2033  sei();
2034  GENERATOR generator;
2035  TonePlay* play = music;
2036  while (true)
2037  {
2038  TonePlay tone;
2039  EEPROM::read(play, tone);
2040  if (tone.tone == Tone::END)
2041  break;
2042  generator.tone(tone.tone, tone.ms);
2043  ++play;
2044  }
2045 }
2046 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
2047 This example plays a short melody, stored in EEPROM, to a buzzer wired on Arduino UNO D9 pin.
2048 It uses FastArduino `devices::audio::ToneGenerator` support to generate square waves at the required
2049 frequencies, in order to generate a melody intro that Star Wars fan will recognize!

2082 The advantages of storing the melody in EEPROM are:
2083 - it can easily be changed to another melody without any change to the program (flash)
2084 - it does not use Flash memory for storage
2085 - it does not use SRAM memory in runtime (although it might, if entirely read from EEPROM
2086 to SRAM, but why would you do that?)

2089 specific device, called an ISP programmer. This is beacuse some Arduino bootloaders do not support EEPROM upload.

2091 The important part of the program is the loop where each note is read for EEPROM before being played:
2092 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}

2099 In this snippet, `play` is used just as an "index" (an address actually) to the current note
2100 in `music` array. Beware that `play` cannot be used directly in your program because it points nowhere
2101 actually, really.

2104 in EEPROM, reads the EEPROM content at this location and copies it into `tone`.
2105

2107 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}

2109  }

2113 it is going to play, it it not possible to use a `for` loop with an upper boundary as we could do if `music` was
2114 directly in SRAM.

2117 just a marker of the end of the melody; we check it for every tone read from EEPROM:
2118 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
2119  EEPROM::read(play, tone);

2121  break;

2124 All this is all good but how do we upload the melody to the EEPROM the first time?
2125 
2126 FastArduino make system takes care of this:
2127

2129 with all content to be uploaded to EEPROM: this is based on all variables defined with attribute `EEMEM` in your

2134 all this code, with FastArduino, to play a sequence of tones: you may use `TonePlayer` class
2135 that odes it all for you, and can handle arrays stored in SRAM, EEPROM or Flash memory.

2138 ### Writing content to EEPROM
2139 
2140 There are also programs that may need to store content to EEPROM during their execution. This is possible
2141 with FastArduino EEPROM support.
2142 
2143 Please do note, however, that the number of writes an EEPROM can support is rather limited (100'000 as per
2144 AVR MCU datasheet), hence you should refrain from writing too many times to the EEPROM.
2145 
2146 Also, do note that writing a byte to the EEPROM is not the same as writing a byte to SRAM, this is much slower
2147 (typically between 1.8ms and 3.4ms).

2230 We will focus on EEPROM sections:
2231 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
2232 static const uint8_t MAX_LEN = 64;
2233 char wifi_name[MAX_LEN+1] EEMEM = "";

2236 Here we define `wifi_name` and `wifi_password` as strings of 64 characters maximum (+ terminating `'\0'`),
2237 stored in EEPROM and both "initialized" as empty strings.
2238 
2239 Initialization here does not mean that these variables will automatically be empty string the first
2240 time this program is executed. It means that after `make build` is invoked, an `.eep` file is
2241 generated in order to be uploaded to EEPROM, and that file contains both variables as empty strings.
2242 Hence, provided you have uploaded that `.eep` file to your MCU EEPROM, both variables will then
2243 be empty strings when first read from EEPTOM by your program.
2244 
2245 In the `main()` function, we then read both variables from EEPROM into local variables:
2246 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
2247  char wifi[MAX_LEN+1];
2248  EEPROM::read(wifi_name, wifi, MAX_LEN+1);
2249  char password[MAX_LEN+1];

2251 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
2252 Here we use `EEPROM::read(uint16_t address, T* value, uint16_t count)` function template,
2253 instantiated with `T = char`: we have to specify the count of characters to be read, i.e. the maximum
2254 allowed string size, plus the null termination.
2255 
2256 As in the previous example, we use `wifi_name` and `wifi_password` variables to provide the address
2257 in EEPROM, of the location of these 2 strings.
2258 
2259 Later in `main()`, if the user decided to enter a new WIFI netwrok name and pasword, those are
2260 written immediately to EEPROM:
2261 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
2262  EEPROM::write(wifi_name, wifi, strlen(wifi) + 1);
2263  EEPROM::write(wifi_password, password, strlen(password) + 1);
2264 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
2265 The `EEPROM::write()` function arguments are similar to `EEPROM::read()` arguments.
2266 
2267 Note that here, we do not have to write the full string allocated space to the EEPROM but only the currently
2268 relevant count of characters, e.g. if `wifi` was input as `"Dummy"`, we will write only `6` characters
2269 for `wifi_name` in EEPROM, i.e. the actual string length `5` + `1` for null termination.
2270 This is particularly improtant to do this as writing bytes to EEPROM is very slow (up to ~4ms) so we want to limit
2271 that writing time to the strict minimum necessary.
2272 
2273 Because writing content to the EEPROM is a very slow operation, there are situations where you do not want
2274 to "stop" your program running because it is waiting for `EEPROM::write()` operations to complete.
2275 
2276 This is the reason why `eeprom` namespace also defines a `QueuedWriter` class that uses interruptions to
2277 write content to EEPROM, allowing your `main()` function to run normally during EEPROM writes. For more
2278 details, please check the API.
